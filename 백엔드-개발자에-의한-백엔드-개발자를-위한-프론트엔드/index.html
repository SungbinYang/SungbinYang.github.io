해당 포스팅은 인프런의 백엔드 개발자에 의한, 백엔드 개발자들을 위한 프론트엔드 강의 - 기본편를 참조하여 만들었습니다.

소개

해당 시리즈 포스팅을 왜 작성하려고 하는지 알아보자. 현재 필자는 백엔드 개발자이다. 그런데 백엔드 개발자가 프론트엔드 지식을 알아야 할까? 혹은 백엔드 개발자여서 보여줄 수 있는게 API 문서밖에 없는데 포트폴리오는 어떻게 작성해야 할까? 이는 백엔드 개발자라면 한 번쯤 고민해봤을 법한 질문이다.

가장 흔한 답변은 본인이 만든 API를 서비스화 해보라는 것이다. API 문서만으로는 "이 사람이 무엇을 만들었는지"를 보여주기 어렵지만, 실제로 동작하는 화면이 있으면 그 뒤에 있는 백엔드 설계까지 자연스럽게 드러나기 때문이다. 물론 백엔드 개발자이기에 프론트엔드가 엄청 뛰어날 필요는 없다. 기본적으로 동작만 하면 되는 것이다.

또한, 실무에서는 어드민 사이트나 백오피스를 만들 때 기본적으로 백엔드 개발자가 담당하는 경우가 많다. 이런 도구들은 외부 사용자에게 노출되지 않기 때문에 전담 프론트엔드 개발자를 투입하기 애매한 경우가 많고, 그만큼 백엔드 개발자가 직접 화면까지 만들어야 하는 상황이 자주 발생한다. 이를 위해 백엔드 개발자도 어느정도의 기본적인 프론트엔드 지식을 아는 것이 중요하다. 그러면 단순히 이런 이유로 백엔드 개발자가 프론트엔드 지식을 알아야 할까? 그것만은 아니기에 좀 더 자세한 이유를 알아보도록 하자.

백엔드 개발자가 프론트엔드를 알면 좋은 이유

지금부터 백엔드 개발자가 프론트엔드 지식을 알면 좋은 이유를 상세히 알아보도록 하겠다. 해당 내용은 필자의 주관적인 의견이 많이 들어가져 있으니 그냥 이렇구나라고 받아들이고 이해가 안 되는 것은 그냥 흘려들어도 좋을 것이다.

모든 백엔드 개발자가 프론트엔드 개발자와 일하는 것은 아니다.

모든 백엔드 개발자가 프론트엔드 개발자와 일하는 것은 아니다. 필자도 처음에는 어떤 서비스를 프론트엔드 개발자와 백엔드 개발자가 같이 만들어나간다라고 막연히 생각했다. 하지만 실제 현업에 들어와 보면 모든 서비스가 그런 구조로 굴러가는 것은 아니다. 작은 규모의 팀이나 스타트업같은 경우에는 백엔드 개발자가 프론트 업무도 같이 하는 풀스택 업무를 겸하는 경우가 흔하다. 인력이 한정되어 있기 때문에 한 사람이 여러 영역을 커버해야 서비스가 굴러가기 때문이다. 그리고 이런 상황은 규모가 있는 회사에서도 종종 발생한다. 대기업이라고 해서 모든 백엔드 개발자가 프론트엔드 개발자와 1:1로 짝을 이뤄 일하는 것은 아니다.

일반적으로 어떤 서비스에 대하여 클라이언트와 뭔가 접점이 있는 서비스를 개발해야 한다면 프론트엔드 개발자와 같이 일하기 마련이다. 일반 사용자가 접하는 화면은 디자인, UX, 반응형 대응 등 전문적인 영역이 많기 때문에 전담 인력이 붙는 것이 자연스럽다. 그러나 이러한 서비스를 유지하기 위해서는 내부적으로 훨씬 많은 작업들이 이뤄져야 한다. 예를 들어 고객이 사용하는 서비스 뒷단에 여러 내부 서비스들이 엮여져 있는 경우가 대표적이다. 운영자가 고객 문의를 처리하기 위한 CS 도구, 서비스 상태를 모니터링하는 대시보드, 데이터를 조회하거나 보정하기 위한 내부 툴, 배치 작업을 수동으로 트리거하기 위한 관리자 페이지 같은 것들이 여기에 해당한다.

이런 뒷단의 서비스들은 API 단독으로 이루어질 때도 있지만 때로는 운영자들이 사용하는 UI를 가지고 있기도 하다. 그리고 이런 내부 서비스들의 UI는 프론트엔드 개발자가 개발하기 보다는 백엔드 개발자들이 개발을 진행하고 유지보수 하는 경우가 많다. 외부 사용자를 대상으로 하지 않기에 화려한 디자인이 요구되지 않고, 그보다 "내가 필요한 기능을 내가 가장 잘 안다"는 이유로 백엔드 개발자가 직접 만드는 것이 효율적이기 때문이다. 만약 우리가 이런 백오피스같은 것을 만든다고 생각해보면, 프론트엔드를 어느 정도는 할 줄 알아야 한다는 생각이 들 것이다.

사용자가 어떤 형태로 서비스를 사용하는지 고려하지 않으면 좋은 API를 만들기 어렵다.

다음으로 사용자가 어떤 형태로 서비스를 사용하는지 고려하지 않으면 좋은 API를 만들기 어렵다. 예를 한번 들어보자. 만약 아래와 같이 우리가 사용자에게 서비스에서 발생한 이벤트에 대한 알림을 보여준다고 가정해보자.

그런데 이런 알림 기능이 사용자들이 사용하기에 편한 형태가 되려면 어떻게 해야할까? 보통은 어떤 알림이 오면 해당 알림이 어떤 내용인지 확인하려고 클릭을 할 것이다. 그러면 해당 알림을 만들때는 관련된 내용에 대한 링크가 걸려있어야 할 것이다. 만약 이런 것을 생각하지 않는다면 링크는 제외하고 알림 목록만 뿌려줄 가능성이 있을 것이다. 예를 들어 응답으로 { id, message, createdAt } 정도만 내려주고 끝나는 경우이다. 이런 경우 프론트엔드에서는 "이 알림을 클릭했을 때 어디로 이동시켜야 하지?"라는 문제에 부딪히게 되고, 결국 API를 다시 수정해달라는 요청이 돌아오게 된다. 만약 사용자가 해당 알림 기능을 어떻게 사용할지를 고민했다라면 API 설계 단계부터 targetUrl이나 resourceType, resourceId 같은 필드를 함께 고려하여 개발을 진행했을 것이다.

이것은 단순히 필드 하나를 누락한 문제가 아니다. API는 결국 "어떤 화면에서 어떻게 쓰일 것인가"라는 맥락 위에서 설계되어야 한다. 페이지네이션을 예로 들어도 마찬가지다. 무한 스크롤 방식의 화면이라면 커서 기반 페이지네이션이 적합하지만, 페이지 번호로 이동하는 화면이라면 오프셋 기반이 더 자연스럽다. 프론트엔드가 어떤 UI를 그릴지 전혀 고려하지 않은 채 API를 설계하면, 둘 사이에 끊임없는 조율과 재작업이 생길 수밖에 없다. 결국 "사용자 관점에서 어떻게 쓰일까"를 상상할 수 있는 능력은 좋은 API를 만드는 데 있어 핵심적인 자질이 된다.

백엔드 개발자를 뽑고 있지만 입사 후, 프론트엔드를 해야할 수도 있는데 누굴 뽑아야 할까?

만약 지원자 2명이 있을 때 한 분은 백엔드 개발만 할 줄 알고 한 분은 프론트엔드 개발도 어느정도 할 수 있다고 해보자. 물론 두 지원자의 백엔드 역량은 동일하다고 보는 것이다. 그러면 채용담당자 입장에서는 후자를 채용할 가능성이 높을 것이다. 당장 실무에서 백엔드 개발자가 백오피스와 같은 작업을 진행할 때 프론트 기술도 어느정도 필요하기 때문이기도 하고, 나중에라도 프론트로 간단한 작업을 맡겼을 때 수용적으로 받아들이냐 아니냐의 문제이기도 하기 때문이다.

이것은 단순히 "기술 스택을 하나 더 아는 사람이 유리하다"는 이야기가 아니다. 조직 입장에서 보면, 업무 범위가 명확히 고정되어 있는 경우보다는 그때그때 필요한 일을 유연하게 해낼 수 있는 사람이 훨씬 가치 있다. 실제로 스타트업이나 작은 조직에서는 "이건 내 일이 아닙니다"라는 경계선이 명확한 사람보다, 필요한 일에 손을 뻗을 수 있는 사람을 선호하는 경향이 강하다. 프론트엔드 경험이 있다는 것은 단순히 화면을 그릴 줄 안다는 의미를 넘어서, 업무 경계에 대한 태도가 유연하다는 신호로 읽히기도 한다. 본인의 주력 분야를 확실히 다지되 인접 분야에도 최소한의 이해를 갖춘 인재는, 어떤 조직에서든 환영받기 마련이다.

프론트엔드 개발까지 할 줄 알아야 혼자서 온전한 서비스를 만들 수 있다.

우리는 왜 개발을 할까? 결국 누군가 사용할 서비스를 만들기 위해 개발을 한다. 그런데 백엔드만으로는 사용자에게 닿는 서비스를 완성할 수 없다. 아무리 잘 설계된 API가 있어도 사용자가 그것을 직접 curl로 호출하지는 않기 때문이다. 사용자에게 닿는 마지막 한 걸음은 결국 화면이고, 그 화면을 만들 수 없다면 아이디어가 있어도 혼자서는 서비스화까지 끌고 가기가 어렵다.

개인적으로는 하나의 온전한 서비스를 만들어 본 경험이 개발자로서 성장하는 과정에서 큰 동기부여가 될 것이다. 기획 단계에서 "어떤 문제를 풀 것인가"를 고민하고, 데이터 모델을 설계하고, API를 만들고, 그 위에 화면을 붙여 실제로 누군가 사용할 수 있는 형태로 배포까지 해보는 경험은 백엔드 한 영역만 파고들어서는 얻기 어려운 시야를 열어준다. 또한 전체 흐름을 한 번이라도 경험해본 사람은, 이후 팀으로 일할 때도 다른 역할의 동료가 왜 그런 고민을 하는지 훨씬 잘 이해할 수 있게 된다.

포트폴리오 관점에서도 마찬가지다. "이런 API를 만들었습니다"라는 설명보다, "이런 문제를 풀기 위해 이런 서비스를 만들었고, 그 과정에서 이런 기술적 선택을 했습니다"라는 서사가 훨씬 강한 설득력을 가진다. 화면이 있는 서비스 하나를 온전히 만들어본 경험은, 단편적인 기술 스택 나열보다 훨씬 많은 이야기를 담을 수 있다.

HTTP

마지막으로, 백엔드 개발자가 프론트엔드를 어느정도 알게 되면 자연스럽게 HTTP라는 규약에 대한 이해가 깊어진다. 백엔드 입장에서 HTTP는 주로 "요청을 받아 응답을 돌려주는" 관점으로만 다뤄지기 쉽다. 컨트롤러에서 요청을 매핑하고, 서비스 로직을 실행하고, DTO로 응답을 내려주는 흐름이 거의 전부이다. 하지만 프론트엔드 쪽에서 HTTP를 다뤄보면 이야기가 달라진다.

브라우저의 개발자 도구를 열어 Network 탭을 들여다보면 하나의 페이지가 로딩되기까지 얼마나 많은 요청이 오가는지, 각 요청이 어떤 헤더를 들고 가는지, 쿠키는 어떻게 주고받는지, CORS 정책에 의해 어떤 요청이 차단되는지 같은 것들이 한눈에 드러난다. 이런 경험은 백엔드 개발자에게 굉장히 실용적인 자산이 된다. 이슈가 생겼을 때 "서버 로그만 보고 추측"하는 것이 아니라 "클라이언트에서 어떤 요청이 어떻게 나갔는지" 를 직접 확인할 수 있기 때문이다.

예를 들어 인증/인가 이슈, CORS 관련 오류, 쿠키의 SameSite 속성 문제, preflight 요청(OPTIONS)이 실패하는 상황 등은 서버 로그만 들여다봐서는 원인을 파악하기 어려운 경우가 많다. 프론트엔드 관점에서 HTTP 흐름을 한 번 훑어본 개발자는 이런 문제들을 훨씬 빠르게 진단할 수 있다. 또한 Content-Type, Authorization, Cache-Control 같은 헤더들이 실제로 어떤 식으로 쓰이는지, 상태 코드(4xx, 5xx)가 클라이언트 입장에서 어떻게 해석되고 사용자 경험에 어떤 영향을 주는지도 체감할 수 있다.

결국 프론트엔드 지식은 백엔드 개발자에게 있어 "다른 영역의 기술"이라기보다, 자신이 이미 다루고 있는 HTTP라는 규약을 반대편에서 바라보게 해주는 창문에 가깝다. 그 창문을 한 번 열어본 개발자는, 이후에 API를 설계할 때도 이전과는 다른 감각으로 접근하게 될 것이다.
