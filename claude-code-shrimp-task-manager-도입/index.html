Shrimp Task Manager 설치

로드맵으로 큰 그림을 그려뒀으니, 이제 각 단계를 작은 작업으로 잘게 나눠 안전하게 위임할 차례다. 앞서 정한 분업을 떠올려 보자 — 굵직한 방향은 개발자(로드맵)가, 세분화는 도구가 맡는다. 그 세분화 도구 가 바로 Shrimp Task Manager 다. 본격적으로 쓰기 전에, 먼저 설치 부터 한다.

Shrimp Task Manager란 — 분해하고, 기억한다

Task Master·Shrimp는 그냥 할 일 목록 이 아니다. AI 기반 지능형 작업 관리 도구 로, 두 가지를 해준다.

쉽게 말씀드리면 우리가 개발해야 할 사항을 관리 가능한 작업으로 분해 를 하고, 클로드 코드를 이용해서 개발을 할 때 — 어제 작업했던 내용을 오늘도 참고하고 내일도 참고 할 수 있도록 *세션 간

맥락을

기억* 한다는 거예요.

능력 설명
작업 분해 복잡한 프로젝트 를 관리 가능한 작업 단위로 쪼갠다
컨텍스트 유지 세션이 바뀌어도 어제의 맥락 을 기억

즉 큰 작업을 잘게 나누면서, 세션을 넘나드는 기억 까지 갖춘 셈이다. 이걸 MCP 서버로 클로드 코드에 물려준다.

1단계 — 클론하고 빌드하기

설치는 소스를 직접 받아 빌드하는 방식이다. GitHub 에서 "쉬림프 태스크 매니저(Shrimp Task Manager)" 를 검색해 공식 저장소 를 찾은 뒤, 아래 순서대로 진행한다.

# ① 저장소 복제
git clone <shrimp-task-manager-repo-url>

# ② 프로젝트 디렉토리로 이동
cd mcp-shrimp-task-manager

# ③ 의존 라이브러리 설치
npm install

# ④ 빌드
npm run build

npm run build 까지 끝나면 Shrimp 본체 는 준비 완료다. 이제 이걸 클로드 코드와 연결 하는 일만 남았다.

2단계 — MCP 서버로 연결하기 (mcp.json)

Shrimp는 MCP 서버로 동작한다. 공식 문서의 mcp.json 설정 을 복사해 프로젝트에 붙여넣고, 핵심 값들 만 내 환경에 맞게 채운다.

{
  "mcpServers": {
    "shrimp-task-manager": {
      "command": "node",
      "args": [
        "<설치한 Shrimp 폴더 경로>/dist/index.js"
      ],
      "env": {
        "DATA_DIR": "<태스크 데이터를 저장할 디렉토리 경로>",
        "TEMPLATES_USE": "en",
        "GUI_ENABLED": "true"
      }
    }
  }
}

각 값의 의미는 다음과 같다.

설정 값 역할
args 설치한 Shrimp 폴더 경로 를 지정 — 어떤 프로그램을 실행할지
DATA_DIR 작업 목록·메모리 등 관리 데이터 가 저장될 디렉토리
TEMPLATES_USE 작업 관리 템플릿 언어 — en(영어)·zh(중국어), ko(한국어) 커스터마이징 가능
GUI_ENABLED true 로 두면 웹 기반 GUI 인터페이스 활성화

특히 DATA_DIR 에서 헤매는 경우가 많다. 프로젝트별로 둘지 공통 디렉토리 로 둘지 선택의 문제인데, 강사도 이 지점을 콕 집어준다.

이러한 디렉터리를 설정하는 것 도 헷갈려 하시는 분들 이 계시더라고요.

그럴 땐 어렵게 생각하지 말고, 데이터를 담을 폴더를 하나 만들어 그 경로를 적어주면 된다.

그러면 해당 디렉터리에 폴더를 만들게요.

3단계 — 설치 확인

연결이 제대로 됐는지는 claude mcp list 한 줄로 확인한다. 목록에 shrimp-task-manager 가 보이면 설치 성공 이다.

claude mcp list

정리하며

Shrimp Task Manager 설치 과정을 정리하면 다음과 같다.

  • 무엇? → 작업을 분해 하고 세션 간 맥락을 기억하는 AI 작업 관리 도구
  • 1단계 → git clone → npm install → npm run build 로 본체 빌드
  • 2단계 → mcp.json에 args(경로) · DATA_DIR(데이터) · TEMPLATES_USE · GUI_ENABLED 설정
  • 3단계 → claude mcp list 로 연결 확인
  • 팁 → DATA_DIR 이 헷갈리면, 데이터용 폴더를 하나 만들어 경로로 지정

설치는 연결까지 일 뿐, 진짜는 지금부터다. 다음 챕터에서는 이 Shrimp로 로드맵의 각 단계를 초기화하고 — 실제 작업 계획 을 세워, 본격적인 **구현 ** 으로 들어가 보자.

Shrimp 초기화 및 작업 계획 세우기

설치를 마쳤으니, 이제 진짜다. 큰 그림(로드맵) 을 작은 작업(서브 태스크) 으로 잘게 나누는 일 — 그 흐름은 ① 초기화 → ② 작업 계획 → ③ 심층 분석·분할 세 박자로 진행된다.

1단계 — 프로젝트 초기화 (init project rules)

가장 먼저 할 일은 Shrimp에게 우리 프로젝트를 학습시키는 것이다. 클로드 코드를 열고 init project rules 를 입력한다.

그러면 init project rules를 입력해서 바로 프로젝트를 초기화 해 보도록 하겠습니다.

그러면 Shrimp가 프로젝트 전체 코드를 훑어, 자기가 두고두고 참고할 메모리 파일 을 만든다 — 바로 shrimp-rules.md 다.

그러면 우리 프로젝트 전체 코드를 파악 해서, 태스크 매니저가 참고할 수 있는 메모리 파일 을 생성할 거예요.

이 shrimp-rules.md 가 이후 모든 작업의 기준점 이 된다. 세션이 바뀌어도 맥락을 기억한다던 그 기억의 정체 가, 바로 이 규칙 파일인 셈이다.

2단계 — 초기 작업 계획 (plan_task)

초기화가 끝나면 대화를 /clear 로 한 번 비운다. 깨끗한 컨텍스트 에서 계획을 세우기 위해서다. 그다음 plan_task 로 로드맵의 1단계 작업을 계획하라고 요청하되, 로드맵 파일을 컨텍스트로 같이 넣어준다.

plan_task — ROADMAP.md 의 1단계(골격) 작업을 계획해줘
# 이때 docs/ROADMAP.md 를 컨텍스트로 함께 첨부

계획이 서면 list_tasks 로 확인한다. 1단계가 두 개의 작업 으로 나뉘어 있는 걸 볼 수 있다.

도구 역할
plan_task 로드맵 기반 초기 작업 계획 수립
list_tasks 현재 나뉜 작업 목록 을 확인

3단계 — 심층 분석과 서브 태스크 분할

plan_task 가 끝이 아니다. MCP 스펙 을 열어보면, Shrimp에는 계획 이후 를 위한 도구가 더 있다.

도구 역할
analyze_task 계획된 작업을 더 깊이 분석
reflect_task 분석한 태스크를 검토·개선 — 자동으로 이어 실행
split_tasks 큰 작업을 서브 태스크 로 분할

첫 번째 작업을 골라 analyze_task 로 깊이 분석하면, 흥미롭게도 reflect_task 가 자동으로 이어져 태스크를 검토·개선한다. 강사도 이 똑똑함 에 놀란다.

제가 쪼개려고 했는데, 스스로 똑똑하게 해당 작업까지 수행하고 있네요.

마지막으로 split_tasks 를 실행하면, 첫 번째 작업이 무려 9개의 서브 태스크 로 잘게 쪼개진다. 큰 덩어리 가 낮은 복잡도의 작은 작업들로 변신하는 순간이다.

이처럼 shrimp-task-manager를 이용하면 단순히 태스크를 분리 하는 걸 넘어 — 또다시 분석하고, 검토하고, 서브 태스크를 나누는 등, 유연하게 우리 태스크를 관리할 수 있죠.

💡 분할이 마음에 안 들면 그대로 두지 말자. split_tasks 를 반복 실행 하거나, 클로드 코드 PLAN 모드를 활용해 **작업

계획을

다시** 나눌 수도 있다.

정리하며

Shrimp 초기화와 작업 계획 과정을 정리하면 다음과 같다.

  • ① 초기화 → init project rules → 전체 코드 파악 → shrimp-rules.md(메모리)
  • ② 계획 → /clear → plan_task(로드맵 첨부) → list_tasks 로 확인
  • ③ 분석·분할 → analyze_task → (자동) reflect_task → split_tasks → 9개 서브 태스크
  • 유연함 → 단순 분리를 넘어 분석·검토·재분할 → 마음에 안 들면 PLAN 모드로 다시
  • 핵심 → 큰 덩어리 를 작고 안전한 작업 으로 → 이제 하나씩 구현하면 된다

로드맵의 큰 그림 이 실행 가능한 서브 태스크 로 모두 내려왔다. 이제 남은 건 하나씩 처리 하는 일 — 다음 챕터부터는 이 잘게 나뉜 작업들을 따라 본격적인 구현 에 들어가 보자.

작업 계획하기 및 실행하기 (plan task, execute task)

작업을 잘게 나눴으니, 이제 그 작업을 하나씩 실행 할 차례다. Shrimp 개발의 기본 루프 는 단순하다 — plan_task( 계획) → execute_task(실행). 이 한 사이클을 처음부터 끝까지 따라가 본다.

어디서 시작하나 — 로드맵이냐, PRD냐

plan_task 는 해야 할 일 을 작은 단위로 쪼개 계획을 세워준다. 그런데 무엇을 입력으로 주느냐는 프로젝트 상황 에 따라 갈린다.

상황 시작점
복잡한 프로젝트 큰 로드맵을 먼저 → plan_task 로 세부 작업 으로
개발 경험이 적을 때 PRD 기반으로 바로 plan_task

우리는 전자 를 택했다. 앞서 로드맵을 만든 이유가 바로 여기에 있다.

복잡한 프로젝트인 경우에는 태스크 관리 도구에게 작게 쪼개서 위임 할 수 있다라는 예시를 보여드리기 위해, 제가 로드맵을 생성 했습니다.

물론 정답이 하나 인 건 아니다.

하지만 이러한 건 운영하시는 분들이나 프로젝트 상황에 따라 달라질 수 있습니다.

계획 — plan_task + 로드맵 참조

이제 plan_task 로 "애플리케이션 골격 구축" 같은 업무를 요청하되, 로드맵 파일을 파일 참조로 함께 넣어준다. 그러면 Claude가 프로젝트를 분석 해 첫 번째 작업 부터 계획하기 시작한다.

그러면 Shrimp Task Manager의 plan_task 기능을 이용해서, 첫 번째 작업을 계획 하는 걸 확인할 수 있고요.

확인 — list_tasks와 Web GUI

계획이 서면 두 가지 방법으로 작업 목록 을 확인한다.

방법 확인 방식
list_tasks 터미널 에서 텍스트 로 작업 목록 확인
Web GUI shrimp_data 의 WebGUI.md → 로컬 서버 주소 → 시각적 으로 확인

특히 Web GUI 가 편하다. 설치 때 GUI_ENABLED를 켜뒀다면, shrimp_data 디렉터리의 WebGUI.md 에 적힌 로컬 서버 주소 로 들어가, 작업 목록을 한눈에 볼 수 있다. 각 작업을 클릭 하면 상세 설명 까지 펼쳐진다.

실행 — execute_task

계획을 확인했으면, 드디어 execute_task 로 작업을 수행한다. 작업 명칭 이나 ID 를 입력하면, Task Manager가 해당 작업을 단계별로 진행한다.

그러면 바로 진행 해 보도록 할게요.

execute_task — 1번 작업을 수행해줘
# 작업 명칭 또는 ID 로 지정

이번 첫 작업은 Notion API 설정을 위한 환경 변수 구성 이었고, 무사히 완료됐다. 실제 Notion API 키 설정 은 * 다음 단계* 로 이어진다.

더 나아가기 — continuous_mode와 research

plan_task → execute_task 가 기본 루프 지만, Shrimp에는 상황별 무기가 더 있다.

모드 언제 쓰나
continuous_mode 작업을 하나씩 말고 — 모든 작업을 자동 으로 연쇄 실행하고 싶을 때
research 개발 전 학습·연구 가 필요할 때 — 버그 수정 계획, 기술 스택 학습, 도메인 지식 습득

즉 기능 개발 만이 아니라, 버그 수정 의 작업 계획을 짜거나 낯선 기술 을 먼저 조사하는 데까지 — Shrimp를 유연하게 활용할 수 있다.

정리하며

작업 계획·실행 과정을 정리하면 다음과 같다.

  • 기본 루프 → plan_task(계획) → execute_task(실행)
  • 시작점 → 복잡하면 로드맵, 경험 적으면 PRD 기반
  • 계획 → plan_task + 로드맵 파일 참조 → 첫 작업부터 계획
  • 확인 → list_tasks(텍스트) + Web GUI(WebGUI.md → 로컬 서버)
  • 실행 → execute_task + 작업명/ID → 단계별 진행 → Notion API 환경 변수 완료
  • 확장 → 전체 자동 은 continuous_mode, 학습·연구 는 research

이렇게 큰 로드맵 에서 출발해 작은 작업 하나 를 완료하기까지, 계획 → 실행 의 한 사이클을 돌았다. 다음은 방금 깔아둔 환경 변수 위에 실제 Notion API를 얹어, 진짜 기능 을 붙여볼 차례다.
