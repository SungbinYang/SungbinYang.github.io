Notion API 키 및 Database ID 설정

앞선 챕터에서 무엇을 만들지 정하고 PRD 까지 다듬었다. 이제 코드를 짜기 전 에 빠뜨리면 안 되는 밑작업 이 있다 — 견적서를 담을 Notion 데이터베이스 를 만들고, 웹이 그 데이터에 접근할 열쇠 를 준비하는 일이다.

[웹 애플리케이션]
      │  API 키(열쇠) + Database ID(주소)
      ▼
[Notion 데이터베이스 = 견적서·품목]

순서는 이렇다. ① API 키 발급 → ② 데이터베이스 생성 → ③ 통합 연결 → ④ 속성 설정 → ⑤ 샘플 데이터 → ⑥ Database ID 추출. 하나씩 따라가 보자.

① API 키 발급 — Notion이라는 금고의 열쇠

먼저 웹이 Notion에 접근할 권한 이 필요하다. 강사의 비유가 명쾌하다.

이러한 API Key는 우리 견적서 데이터베이스를 만들 거거든요. Notion에 이러한 데이터베이스의 애플리케이션이 접근할 수 있게 하는 Key 라고 보시면 됩니다.

키를 발급받는 곳은 My Integrations 다. 사이트를 모르겠다면 Notion API 키 로 검색해도 된다. 다만 Notion 홈페이지가 자주 바뀌어 화면이 헷갈릴 수 있는데, 강사가 최신 절차 를 따로 안내한다.

  1. app.notion.com/developers/connections 로 접속
  2. 우측 상단 [신규 연결] 클릭
  3. 연결 이름 은 자유롭게, 인증 방법은 [액세스 토큰] 선택 → [연결 생성하기]
  4. [기능] 에서 필요한 권한 체크 → 발급된 액세스 토큰 확보

만약에 이러한 키를 통해서 노션 데이터베이스의 값을 추가한다든가 수정 을 원하시는 분들은 모두 체크 하면 될 거예요.

발급받은 키는 코드에 직접 박지 말고 — 프로젝트의 .env.local 파일에 넣어둔다. 비밀 키 는 절대 깃에 올라가면 안 되니, 환경 변수로 분리하는 게 기본이다.

② 데이터베이스 생성 — Invoice 와 Items

키를 손에 쥐었으니, 이제 데이터를 담을 그릇 을 만든다. 견적서 시스템에는 두 개 의 데이터베이스가 필요하다.

데이터베이스 역할
Invoice 견적서 한 건 의 정보
Items 견적서에 들어가는 품목들

Notion 페이지에서 슬래시(/) 명령어로 데이터베이스 를 두 개 만들면 된다. 왜 둘로 나누나? — 견적서 하나에 여러 품목 이 붙기 때문이다. 견적서와 품목을 한 테이블에 욱여넣으면 관리가 지저분해진다. 그래서 견적서 / 품목 을 나누고, 나중에 관계 로 묶는다.

③ 통합 연결 — 많은 분들이 실수하는 지점

데이터베이스를 만들었다고 바로 접근 되는 게 아니다. ①에서 만든 API 통합 을 각 데이터베이스에 연결해줘야 한다.

  • 데이터베이스 우측 상단 더보기(···) → [연결] → 만들어둔 통합 선택

여기서 강사가 못을 박는다.

많은 분들이 실수를 하세요.

무슨 실수냐 — 페이지가 아니라 데이터베이스 에서 연결을 걸어야 한다는 점이다. 겉보기엔 비슷한 화면이라, 페이지에 연결해두고 "왜 접근이 안 되지?" 하며 헤매기 쉽다. Invoice 와 Items 둘 다 잊지 말고 연결하자.

④ 속성 설정 — Claude Code에게 물어본다

각 데이터베이스에 어떤 컬럼(속성) 을 둘지 정할 차례다. 그런데 혼자 고민할 필요 없다.

이러한 것도 Claude Code에게 물어보도록 할게요.

프로젝트 맥락(PRD·CLAUDE.md)을 이미 파악한 클로드에게 "이 견적서 프로젝트에 맞는 Notion 속성을 설계해줘" 라고 하면 * 가이드* 를 받을 수 있다. 대략 이런 모양이 나온다.

  • Invoice → 견적서 번호·클라이언트 명·발행일·유효기간·상태·총금액·항목
  • Items → 항목명·수량·단가·금액·견적서(연결)

설정에서 두 가지 포인트 가 핵심이다.

  1. 수식(Formula) — Items 의 금액 은 직접 입력하지 않고 수량 × 단가 로 자동 계산

    그러면 AI 기능으로 인해서 자동으로 수량 곱하기 단가 가 설정되었고요.

  2. 관계형(Relation) — Invoice 의 항목 속성을 Items 와 관계형으로 연결 → 양방향 관계 설정

양방향 이 중요하다. 견적서에서 "이 견적서에 딸린 품목들" 을 보고, 품목에서도 "내가 속한 견적서" 를 거꾸로 볼 수 있게 된다. 속성은 정답이 없다 — 강사 말처럼 우리 입맛에 맞게 더하고 빼면 된다.

⑤ 샘플 데이터 삽입 — 빈 그릇은 테스트가 안 된다

구조를 잡았으면, 실제로 잘 도나 확인하려고 샘플 데이터 를 넣는다.

  • Invoice → 견적서 번호·발행일·유효기간·총금액 등 한 건 입력
  • Items → 웹사이트 디자인·로고 제작·명함 디자인 같은 품목을 추가하고, 각 항목을 해당 견적서에 연결

이때 금액이 수량 × 단가 로 알아서 채워지고, 견적서와 품목이 관계로 묶이는지 눈으로 확인하면 — 4단계 설정이 제대로 됐다는 증거 다.

⑥ Database ID 추출 — 쿼리 스트링을 떼라

마지막. 웹이 "어느 데이터베이스를 읽어야 하는지" 알려줄 주소(Database ID) 가 필요하다.

  1. Invoice 데이터베이스의 공유 링크 복사
  2. URL에서 쿼리 스트링 (물음표 ? 뒤 전부)을 제거
  3. 남은 부분의 ID 를 .env.local 에 설정
https://www.notion.so/{여기가-Database-ID}?v=abcd1234...
                       └──────┬──────┘ └──────┬──────┘
                         이게 ID         쿼리 스트링 (제거)

많은 분들이 깜빡하시더라고요.

쿼리 스트링까지 그대로 붙여넣으면 ID가 어긋나 접근이 안 된다. ? 뒤를 깔끔히 잘라내는 걸 잊지 말자. 참고로 Items 는 따로 ID가 필요 없다 — Invoice 와 관계형으로 연결 돼 있어, 견적서를 읽으면 딸린 품목 까지 함께 따라오기 때문이다.

정리하며

Notion 연동 준비를 정리하면 다음과 같다.

  • ① API 키 → app.notion.com/developers/connections → [신규 연결]·[액세스 토큰] → .env.local 에 저장
  • ② 데이터베이스 → Invoice(견적서) + Items(품목) 두 개 로 분리
  • ③ 연결 → 통합을 각 DB에 연결 → ⚠️ 페이지 말고 데이터베이스 에서
  • ④ 속성 → Claude Code에게 설계 요청 → 금액=수식, 항목=관계형(양방향)
  • ⑤ 샘플 → 견적서·품목 한 세트 넣어 연결·계산 동작 확인
  • ⑥ Database ID → 공유 링크에서 쿼리 스트링 제거 → .env.local (Items 는 불필요)

여기까지가 코드 한 줄 없이 끝내는 연동의 토대 다. 키(권한)와 ID(주소), 그리고 관계로 묶인 두 데이터베이스 가 준비됐으니 — 다음 챕터부터는 이 위에 실제 웹 애플리케이션 을 올려보자.

서브에이전트: 개발 전문가

Notion 연동이라는 토대 가 깔렸다. 이제 진짜 코드 를 짤 차례다. 그런데 강의는 코딩 자세 부터 바꾸라고 한다 — 당신은 직접 키보드를 두드리는 개발자 가 아니라, 개발 팀장 이다.

우리가 AI 활용을 해서 개발을 한다라고 생각을 하지 말고, '다음 작업을 해야 한다' 라고 생각을 해보세요.

그리고 우리는 현재 개발 팀장의 위치 에 있다라고 가정을 해보도록 하겠습니다.

팀장은 코드를 짜지 않는다. 무엇을 할지 정하고 → 적임자에게 맡기고 → 결과를 검수 한다. 이번 챕터의 흐름이 정확히 그렇다.

일감 꺼내기 — list-tasks 와 execute-task

먼저 지금 할 일 을 확인한다. Shrimp Task Manager로 쪼개둔 작업을 list-tasks 로 꺼내고, execute-task 로 실행한다.

list-tasks   → 할 일 목록 확인 (팀장이 보드를 본다)
execute-task → 작업 실행      (담당자에게 맡긴다)

물론 모든 작업 에 전문가가 필요한 건 아니다.

물론 이러한 작업 같은 경우에는 크게 복잡한 작업은 아니에요. 그래서 execute-task 기능을 통해서 바로 작업을 수행해도 좋지만, 만약에 효과적으로 작업을 수행하지 못한다면 서브 에이전트를 이용해서 이러한 작업을 수행하는 건 매우 효과적 이죠.

즉 간단한 일 은 메인에게 바로 시키고, 막히거나 까다로운 일 이 나오면 — 전문가 를 부른다.

전문가 투입 — Next.js App Router 개발자

이 프로젝트의 기술 스택은 Next.js(App Router) 다. 그래서 그 분야만 파는 서브 에이전트를 하나 만든다.

  • 역할 — Next.js App Router 전문 개발자
  • 컨텍스트 — Next.js 프로젝트 구조 같은 문서를 복사·붙여넣어 전문성을 끌어올림

여기서 핵심은 컨텍스트를 얼마나 잘 먹이느냐 다. 역할 이름만 "전문가" 라고 붙인다고 전문가가 되진 않는다. 공식 문서·구조 규칙 을 직접 넣어줘야 진짜 전문가 가 된다. 그리고 이 방식은 기술 스택을 가리지 않는다.

수강생분들이 어떠한 기술 스택 으로 개발을 하시더라도, 메인 컨텍스트에서 진행을 했을 때 결과가 만족스럽지 않다면, 해당 컨텍스트를 추가하여 전문가 서브 에이전트 를 추가한 다음에, 이러한 서브 에이전트로 다시 작업을 수행 한다면, 좀 더 만족한 결과물 을 얻을 수 있을 거예요.

이게 서브 에이전트의 진짜 쓸모 다. "메인이 못 하면 → 컨텍스트 붙인 전문가로 재시도" — 이 한 줄이 핵심 패턴이다.

토큰 관리 — compact 로 숨 고르기

개발이 길어지면 컨텍스트가 불어나 토큰을 잡아먹는다. 이때 compact 로 대화를 요약 해 토큰을 줄이고, 필요하면 **세션을 닫았다 다시 열어 ** 이어서 진행한다. 다시 들어와도 Next.js 앱 디벨로퍼 에이전트 를 부르면 전문성은 그대로 살아 있다.

💡 작업 순서는 바뀔 수 있다. 다만 의존 관계 는 GUI로 확인한 것과 동일하니, 순서가 좀 달라져도 놀랄 필요 없다.

전문가가 라우팅 구조와 페이지 골격 을 만들어내면 — 견적서·로딩·Not Found·에러 페이지가 한꺼번에 생긴다. 검수 는 팀장의 몫이니, 생성된 코드와 페이지를 눈으로 확인한다.

타입 정의 — Notion 속성을 데이터 모델로

다음은 타입 정의 다. 그런데 맨손 으로 시키지 않는다. 앞서 만든 Notion 데이터베이스의 속성 을 컨텍스트로 넣어준다.

  1. Notion DB 속성 정보를 컨텍스트에 추가
  2. CSV로 내보낸 데이터 를 압축 해제해 실제 형태 파악에 활용
  3. Bypass permissions 모드로 타입 생성 완료

추상적인 타입 이 아니라, 실제 Notion 스키마에 들어맞는 타입이 나온다는 게 포인트다. 굵직한 작업 전엔 Plan mode로 무엇을 할지 미리 확인하면 삽질을 줄일 수 있다.

마무리 — 로드맵 체크, 그리고 다음

한 단계가 끝나면 ROADMAP.md에 완료 체크 를 남긴다. 번거로운 완료 내역 기록 은 덜어내고, 핵심만 깔끔하게 관리한다. 그리고 Plan mode 로 두 번째 단계 를 계획한다 — 이때도 원칙은 하나다.

계획 수립 시 컨텍스트를 최대한 추가 하는 것이 좋다.

로드맵의 다음 칸은 UI/UX 완성 이다. 골격은 섰으니, 이제 보기 좋게 입힐 차례 — 그건 UI/UX 전문 서브 에이전트 의 일이다.

정리하며

개발 전문가 서브 에이전트를 정리하면 다음과 같다.

  • 관점 → 내가 코딩 이 아니라 → 개발 팀장 으로 "다음 작업" 을 지휘
  • 흐름 → list-tasks → execute-task → 간단하면 메인, 막히면 전문가
  • 전문가 → Next.js App Router 개발자 → 공식 문서·구조를 컨텍스트로 먹여야 진짜 전문가
  • 핵심 패턴 → 메인이 못 하면 → 컨텍스트 붙인 서브 에이전트로 재시도
  • 효율 → compact로 토큰 절약 · Plan mode로 사전 확인
  • 타입 → Notion 속성·CSV 를 컨텍스트로 → 실제 스키마에 맞는 타입
  • 관리 → ROADMAP.md 완료 체크 → 다음은 UI/UX

좋은 팀장은 직접 다 하지 않는다. 적임자를 세우고, 맥락을 충분히 쥐여준 뒤, 결과를 검수 한다. 골격이 섰으니 — 다음 챕터에서는 **UI/UX 전문가 **를 불러 화면에 살을 붙여보자.

서브에이전트: UI/UX 전문가

앞 챕터에서 라우팅과 페이지 골격 을 세웠다. 뼈대는 섰지만 — 아직 밋밋한 회색 화면 이다. 이제 보기 좋게 살을 붙일 차례다. 그리고 그 일엔 또 다른 전문가 가 붙는다.

그리고 이번 단계에서는 동적인 기능 없이 UI를 개발 하는 작업이기 때문에

핵심은 이번엔 기능을 안 건드린다 는 점이다. 마크업과 스타일링 에만 집중한다 — 즉 정적인 화면 부터 깔끔하게 완성하는 단계다.

ui-markup-specialist — 마크업만 파는 전문가

개발 전문가와 똑같은 방식으로, UI/UX 전용 서브 에이전트를 만든다.

  • 역할 — UI/UX 마크업 전문가 (ui-markup-specialist)
  • 스택 — Next.js · TypeScript · Tailwind CSS · shadcn/ui
  • 범위 — 동적 기능은 제외, 마크업·스타일링 에만 집중

프롬프트는 '한 번에' 안 끝난다 — 계속 고친다

여기서 강사가 반복해서 강조 하는 습관이 나온다. 처음 만든 에이전트를 그대로 두지 않는다. MCP 서버를 최대한 활용 하도록 Context7·Sequential Thinking 을 붙여 에이전트를 수정 한다.

이처럼 제가 계속 강조드리는 거지만, 이러한 서브 에이전트나 사용하는 프롬프트는 지속적으로 수정해 나간다면 우리가 원하는 결과를 얻는 데 정말 큰 도움 이 되죠.

즉 완벽한 한 방 을 노리지 말고 — 돌려보고 → 부족하면 프롬프트를 고치고 → 다시 가 정석이다. 이게 프롬프트 엔지니어링의 본질이다.

Continuous Mode — 8개 작업을 연달아

이제 Shrimp Task Manager의 Continuous Mode 를 켠다. 작업 목록(8개) 을 순차적으로 자동 실행 하는 모드다.

[Continuous Mode]
작업1 → 작업2 → 작업3 → … → 작업8   (멈추지 않고 연달아)

UI 마크업은 크게 복잡하지 않은 반복 작업이라, ui-markup-specialist 에게 연속으로 맡기기 딱 좋다. 다 돌면 테스트 URL 로 들어가 견적서 UI가 제대로 뜨는지 눈으로 확인한다.

안 끝난 것처럼 보일 때 — 당황 금지

자동으로 길게 돌리다 보면 작업이 끝났는데 대기 중 처럼 보일 때가 있다.

간혹가다 이럴 경우가 있습니다. 그럴 때는 당황하지 마시고요, 하나씩 확인 하시면 돼요.

이럴 땐 Plan mode로 전환해 각 작업이 실제로 구현됐는지 하나씩 점검한다. 전부 100% 완료 되고 요구사항이 충족됐으면, Shrimp의 작업을 모두 완료 로 체크한다. 그래도 빠진 부분(에러·로딩 상태 UI) 이 있으면 — 메인 컨텍스트에서 직접 마저 개발한다.

반복은 커맨드로 — update-roadmap

작업이 끝날 때마다 ROADMAP.md에 완료 체크 를 남기는데, 이걸 매번 손으로 하면 번거롭다. 자주 쓰는 프롬프트 는 * 커스텀 커맨드*로 묶는다.

  1. docs 디렉터리에 update-roadmap.md 생성 → 완료 작업 체크 커맨드 작성
  2. ROADMAP.md 를 컨텍스트로 추가 → Plan mode로 요청
  3. bypass permissions 모드로 커맨드 실행

이렇게 해두면 "로드맵 갱신해줘" 한마디로 자동 정리 된다.

⭐ 꼭 — Git 커밋

마지막으로 강사가 가장 힘주어 당부하는 게 있다.

Git으로 커밋을 하는 건 정말 중요해요.

이유는 분명하다. AI는 코드를 한꺼번에 많이 고친다. 무엇이 어떻게 바뀌었는지 사람이 일일이 따라잡기 어렵고, 그러다 치명적인 상황 이 터질 수 있다. 그래서 기능 하나 완성 → 즉시 커밋 을 원칙으로 삼는다. 되돌릴 지점 을 촘촘히 박아두는 것 — 이게 AI 시대의 안전벨트 다.

정리하며

UI/UX 전문가 서브 에이전트를 정리하면 다음과 같다.

  • 단계 → 동적 기능 제외 → 마크업·스타일링 만 (정적 화면 완성)
  • ui-markup-specialist → Next.js·TS·Tailwind·shadcn/ui 전용 서브 에이전트
  • 개선 → MCP(Context7·Sequential Thinking) 추가 → 프롬프트는 계속 고친다
  • 자동화 → Shrimp Continuous Mode 로 8개 작업 연달아
  • 점검 → 멈춘 듯 보이면 당황 말고 → Plan mode로 하나씩 확인 → 빠진 건 메인에서
  • 커맨드 → update-roadmap로 로드맵 갱신 자동화
  • ⭐ Git 커밋 → 기능 완성마다 즉시 커밋 → AI 시대의 안전벨트

뼈대에 살이 붙어 화면이 제 모습을 갖췄다. 전문가에게 맡기고 → 반복은 커맨드로 → 지점마다 커밋 — 이 리듬이 몸에 배면 개발 속도가 확 붙는다. 다음 챕터에서는 이 정적 화면에 실제 데이터와 동적 기능 을 불어넣어 보자.

개발 워크플로우: UI/UX 개선하기

앞 챕터에서 정적 화면 을 한 번에 찍어냈다. 그런데 처음부터 완벽한 화면 이 나올 리 없다. 강사의 한마디가 모든 걸 요약한다.

하지만 우리가 개발을 진행하다 보면 계획대로 되는 게 있나요?

이번 챕터는 예상 못 한 수정 요구 가 튀어나왔을 때, 그걸 워크플로우 안에서 깔끔히 처리하는 법이다.

왜 더미 데이터로 먼저 만드나

가장 먼저 짚는 원칙. 실제 데이터를 붙이기 전에, 더미 데이터 로 UI/UX를 먼저 완성 한다. 이유가 분명하다.

[더미 데이터로 UI 완성]  → 디자인 수정이 쉽다 (마크업만 손보면 끝)
        ↓ 그 다음
[실제 DB 연동]           → 이제 수정은 코드 전체를 건드린다

데이터를 연동한 뒤 디자인을 고치면, 단순 UI 수정 이 아니라 비즈니스 로직까지 얽힌 복잡한 코드 변경이 된다. 그러니 모양부터 확정 하고, 데이터는 나중에 붙이는 게 영리하다.

핵심 기법 — 스크린샷을 첨부하라

UI를 고칠 때 말로만 설명하면 잘 안 먹힌다. 스크린샷을 붙여서 요청하는 게 훨씬 효과적이다. 이건 사람이 디자인하는 방식 그대로다.

맨 처음에 초안 을 작성하고, 마음에 들지 않는 부분 은 개선을 하고 마음에 드는 부분 은 점진적으로 발전 해가면서 지속적으로 디자인을 발전해가죠.

Anthropic 공식 문서에서도 권장하는 방법 이다. 현재 화면을 캡처 → "이 부분을 이렇게 고쳐줘" → 반복. 이 시각적 피드백 루프를 돌릴수록 클로드의 출력은 눈에 띄게 좋아진다.

[스크린샷 첨부] → [수정 요청] → [결과 확인] → 다시 [스크린샷]
        ↑________________________________________│

갑작스러운 요구는 — plan_task 로

로드맵을 따라가던 중 "컨테이너 폭을 맞춰줘", "간격을 조절해줘" 같은 예상 못 한 요구 가 생긴다. 그럴 땐 즉흥적으로 손대지 말고, Shrimp Task Manager의 plan_task 로 새 작업 계획 을 세운다.

  1. plan_task → 요구사항을 세부 작업으로 분리
  2. list_tasks → 진행 중·대기 중 작업 상태 확인
  3. 순차 진행 요청 → UI 개선 완료

갑자기 끼어든 일 도 계획 → 분리 → 순차 실행 이라는 같은 리듬 으로 처리한다는 게 핵심이다.

Playwright MCP — 클로드가 스스로 화면을 본다

구체적인 수정도 보여준다. 견적서 상세 페이지 의 섹션 너비를 일관되게 맞추는 작업이다. 물론 직접 고쳐도 된다.

개발 경험이 있으신 분들은 이러한 걸 굳이 AI에게 요청을 하지 않고 바로 이런 코드를 수정 할 수 있겠죠.

하지만 여기선 클로드를 최대한 활용 한다. Playwright MCP 서버 를 붙이면 — 클로드가 직접 화면을 띄워 확인 하고, 스스로 수정 계획 을 세운다. 사람이 일일이 스크린샷을 찍어 주지 않아도, 클로드가 눈을 갖는 셈이다. plan_task 와 함께 돌리면 일관된 UI 가 나온다.

정리하며

UI/UX 개선 워크플로우를 정리하면 다음과 같다.

  • 순서 → 더미 데이터로 UI 먼저 → 그다음 실제 DB 연동 (수정 비용 ↓)
  • 핵심 기법 → 스크린샷 첨부 → 초안 → 개선 → 반복 (Anthropic 권장)
  • 갑작스러운 요구 → plan_task로 세부 작업 분리 → list_tasks → 순차 진행
  • 자동 확인 → Playwright MCP → 클로드가 스스로 화면 보고 수정 계획
  • 마인드셋 → 계획대로 안 되는 게 정상 → 워크플로우로 흡수

디자인은 한 방 이 아니라 반복 이다. 더미로 먼저 모양을 잡고 → 스크린샷으로 다듬고 → 갑작스러운 요구는 plan_task 로 — 이 흐름이면 계획이 흔들려도 개발은 흔들리지 않는다. 다음 챕터에서는 이 완성된 화면에 실제 Notion 데이터 를 흘려보내 보자.

핵심 기능 구현하기

앞 챕터까지 화면 은 다 만들었다. 하지만 아직 더미 데이터 가 박혀 있는 껍데기 다. 이제 진짜 알맹이 를 채울 차례다 — 실제 Notion 데이터를 흘려보내고, PDF로 내려받는 핵심 기능이다.

작게 쪼개라 — AI에겐 낮은 복잡도를

기능 구현도 Plan mode로 계획부터 세운다. 그룹 단위로 묶어도 되고, 개별 태스크로 잘게 나눠도 된다. 강사의 원칙은 일관 된다.

AI에게 최대한 낮은 복잡도 를 위임하는 편이에요. 그래야 오류 없이 정확히 구현을 하더라고요.

큰 덩어리 를 한 번에 던지면 클로드가 헤맨다. ROADMAP.md를 참조해 ultrathink 로 작업을 작게 쪼개 두면 — 오류가 확 줄어든다. 이번 단계는 두 갈래 로 나뉜다.

[트랙 A] Notion API 연동 → 실제 견적서 표시
[트랙 B] PDF 생성        → 다운로드 기능

트랙 A — Notion 데이터를 화면에

첫 번째 갈래는 더미 대신 진짜 데이터 를 붙이는 일이다. 여기서 앞서 만든 Notion 연동 전문 서브 에이전트가 다시 등판한다.

  • Notion 데이터베이스 정보 + CSV 파일 을 컨텍스트로 추가
  • 에이전트가 CSV를 분석 → Notion API 통합 구현
  • 브라우저 테스트 로 결과 확인

결과물은 두 가지다. Notion 데이터를 웹 데이터로 변환하는 유틸리티, 그리고 그걸 페이지에 뿌리는 코드. 전문가에게 컨텍스트를 충분히 먹이는 그 패턴 그대로다.

트랙 B — PDF 다운로드

두 번째 갈래는 PDF 생성 패키지 를 깔아 다운로드 기능 을 붙이는 일이다. 진행하다 토큰이 꽉 차면 /clear 로 컨텍스트를 비우고, * execute_task* 로 이어간다.

그런데 이 챕터의 진짜 알맹이 는 따로 있다. 강사가 수없이 데인 끝에 찾은 오류를 줄이는 두 가지 룰 이다.

이전에 제가 AI를 활용했을 때는 "어떠어떠한 걸 구현해줘" 한 다음에 테스트를 했을 때 오류가 정말 많았거든요. 그런데 두 가지를 추가 하고 나서 오류가 정말 줄어들었어요.

⭐ 룰 ① — Playwright로 스스로 테스트하게

첫 번째. 비즈니스 로직을 구현한 뒤, Playwright MCP 통합 테스트 프롬프트를 붙인다. 그러면 클로드가 직접 화면을 띄워 테스트하고, * 버그까지 스스로 잡는다.* 사람이 일일이 눌러보며 확인할 필요가 없어진다.

⭐ 룰 ② — 린트·타입·빌드 체크리스트

두 번째. package.json 의 린트 체크 + 타입 체크 를 구현 직후 돌리게 한다 — 심지어 빌드까지. 자주 쓰는 명령어는 CLAUDE.md에 적어두고, check-all 과 npm run build 로 클로드가 제 손으로 오류를 검출하게 만든다.

그리고 저는 이러한 룰이 Playwright MCP보다 더 강력 하다고 느꼈어요. 특히 작업 완료 체크리스트, 이걸 추가하고 나서 정말 많이 오류가 줄었습니다.

핵심은 "구현 → 자가 검증"을 한 묶음으로 만드는 것이다. 코드를 짜고 끝 이 아니라, 테스트·린트·타입·빌드를 통과해야 완료 — 이 체크리스트가 버그를 입구에서 막는다.

마무리 — 통합 테스트 · 로드맵 · 커밋

두 트랙이 끝나면 통합 테스트 로 마무리한다. list-task로 남은 작업 을 확인하고, 마지막 통합 태스크를 execute-task 로 처리한다. 그다음 루틴 대로다.

  1. /compact 로 컨텍스트 정리
  2. update-roadmap으로 완료 반영
  3. Git 커밋 으로 지점 저장

로드맵을 보면 핵심 기능까지 완료 — 이제 성능 최적화·보안·에러 처리·테스트·배포 만 남았다.

정리하며

핵심 기능 구현을 정리하면 다음과 같다.

  • 계획 → Plan mode·ultrathink로 잘게 쪼개기 → AI엔 낮은 복잡도
  • 트랙 A → Notion 연동 서브 에이전트 + DB 정보·CSV → 실제 견적서 표시
  • 트랙 B → PDF 패키지 → 다운로드 기능 (토큰 차면 /clear)
  • ⭐ 룰 ① → Playwright MCP → 스스로 테스트·버그 수정
  • ⭐ 룰 ② → CLAUDE.md에 check-all·npm run build → 린트·타입·빌드 자가 검증 (강사 왈, Playwright보다 강력)
  • 마무리 → 통합 테스트 → /compact → update-roadmap → 커밋

AI에게 코드를 맡길 때 가장 무서운 건 소리 없는 버그 다. 그래서 작게 쪼개 시키고, 테스트·체크리스트로 스스로 검증 하게 만드는 것 — 이 두 룰이 MVP의 완성도 를 가른다. 다음 챕터에서는 관리자 기능 을 붙이고 실제 배포 까지 달려보자.

핵심 기능 구현 및 배포하기

앞 챕터에서 Notion 데이터 와 PDF 라는 알맹이를 채웠다. 드디어 로드맵의 마지막 칸 — 최적화 와 배포 다. 만든 걸 내 컴퓨터 안 에만 두면 의미가 없다. 이제 세상에 내보낼 차례다.

마지막도 — 계획부터

흐름은 지금까지와 똑같다. ROADMAP.md를 컨텍스트로 넣고 Shrimp Task Manager의 plan_task 로 최적화·배포 작업을 잡는다. 그리고 nextjs-app-developer 서브 에이전트로 execute_task 를 돌린다.

그러면 권장되는 실행 순서 대로 Task 1, 4, 5번 을 먼저 구현하도록 할게요.

번호 순서대로 가 아니라 — 의존 관계가 권장하는 순서 대로 따라가는 게 포인트다.

핵심 최적화 — 서비스를 '운영' 가능하게

연습용 화면과 진짜 서비스 의 차이는 여기서 갈린다. 강사가 붙이는 최적화 항목 들을 보자.

항목 왜 필요한가
캐싱(Caching) 60초간 데이터 재사용 → Notion API 호출 을 줄여 성능 ↑
Rate Limiting 1분당 호출 제한 → 서버 과부하·DDoS 방어, 공평한 서비스
로깅(Logging) 모든 활동·에러 기록 → 문제 터졌을 때 원인 추적
폰트 최적화 불필요한 로딩 제거 → 초기 화면 빠르게
Suspense 점진 렌더링 비동기 로딩 중 UI 표시 → 체감 속도 개선

기능이 도는 것 과 서비스로 버티는 것 은 다르다. 이 항목들이 MVP를 운영 가능한 제품 으로 끌어올린다.

통합 테스트 — 배포 전 마지막 관문

배포 직전, Playwright MCP로 통합 테스트 를 돌린다. 견적서 조회 시나리오 를 짜서 브라우저에서 직접 테스트하고, PDF 다운로드·오류 페이지 까지 점검한다. 클로드가 제 손으로 화면을 눌러보며 검증하니, 사람은 결과만 확인하면 된다.

Vercel 배포 — 세상에 내보내기

이제 Vercel 에 올린다. 흐름은 이렇다.

  1. GitHub에 소스 푸시 (execute_task 가 처리)
  2. vercel.com 로그인 → [Add New] → 현재 프로젝트 임포트
  3. ⭐ 환경 변수 설정 — .env.local 의 API 통합 키·Database ID 를 그대로 등록 + NODE_ENV=production
  4. [Deploy] 클릭 → 빌드·배포

💡 환경 변수를 잊지 말 것. .env.local은 내 컴퓨터에만 있는 파일이라, 깃에 올라가지 않는다. Vercel에 *따로

등록* 하지 않으면 — 배포본은 키도 ID도 없이 텅 빈 채 돌아 접근 오류 가 난다. (많은 분들이 여기서 막힌다.)

빌드가 실패하면 — 로그를 클로드에게

배포가 한 번에 될 거란 보장은 없다. 빌드가 fail 나도 당황할 필요 없다.

빌드가 fail이 발생한다면 에러 로그를 복사 해서 Claude Code에게 해결해 달라고 하면 될 거예요.

LLM은 확률적이라 이슈는 자연스러운 현상 이다. 중요한 건 해결하는 방식 — 에러 로그라는 컨텍스트 를 그대로 던지면, 클로드가 원인을 짚어 고쳐준다. 이게 이 강의가 처음부터 강조해온 컨텍스트 기반 디버깅 이다.

정리하며

핵심 기능 구현과 배포를 정리하면 다음과 같다.

  • 계획 → ROADMAP.md + plan_task → 권장 순서 대로 execute_task
  • 최적화 → 캐싱·Rate Limiting·로깅·폰트·Suspense → 운영 가능한 제품으로
  • 통합 테스트 → Playwright MCP로 조회·PDF·오류 까지 점검
  • Vercel → GitHub 푸시 → 임포트 → ⭐ 환경 변수(+NODE_ENV) → Deploy
  • 빌드 실패 → 당황 말고 → 에러 로그를 클로드에게 → 컨텍스트 기반 해결

배포된 링크로 들어가 견적서가 뜨고 PDF가 내려받아지면 — Notion 기반 견적서 웹 이 완성된 것이다. 마지막으로 로드맵 갱신과 * Git 커밋* 까지 마치면 끝. 돌이켜보면 이 프로젝트의 진짜 수확은 견적서 웹 이 아니라 — **계획 → 전문가 위임 → 자가 검증 → 배포 **로 이어지는 Claude Code AI 워크플로우 그 자체다.
