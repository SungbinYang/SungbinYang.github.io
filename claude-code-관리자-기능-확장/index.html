ROADMAP 버전관리: MVP 이후 이어나가기

지난 시리즈에서 Notion 기반 견적서 웹 의 MVP 를 만들어 배포 까지 마쳤다. 그런데 진짜 일 은 여기서부터다. 강사도 이 질문을 자주 받는다고 한다.

MVP 개발 후에 이후에는 어떠한 워크플로우로 개발을 진행하는지 궁금해 하시더라구요.

결론부터 말하면 — 거창한 게 아니다. AI를 안 쓸 때의 고도화 방식 과 크게 다르지 않다. 로드맵에 단계를 더하거나, 버전을 매겨 이어가면 된다. 이번 챕터는 그 로드맵 버전 관리 의 흐름이다.

무엇을 더 붙이나 — 고도화 요구사항

MVP는 "견적서를 보여주고 PDF로 내려받는" 핵심만 담았다. 이제 살을 붙인다. 강의가 제시하는 고도화 항목은 셋이다.

  • 관리자 레이아웃 — Notion의 견적서 목록 을 관리자 화면 에서 한눈에
  • 링크 복사 — 목록에서 클라이언트에게 보낼 링크 를 바로 복사
  • 다크모드 — 보기 편하게 테마 전환

이러한 로드맵 파일에 다섯 번째 단계 를 추가해서 개발을 진행할 수도 있고요.

여기에 수강생이 원하는 기능 을 자유롭게 더 얹어 테스트해보길 권한다 — 어차피 목적은 견적서 웹 이 아니라 워크플로우 연습 이니까.

핵심 — 로드맵에 '버전'을 매긴다

가장 중요한 대목이다. MVP를 만든 로드맵을 지우고 덮어쓰지 않는다. 버전으로 쌓는다.

docs/
 ├─ prd.md
 └─ roadmaps/
     ├─ roadmap-v1.md   ← MVP 개발에 쓴 로드맵 (보존)
     └─ roadmap-v2.md   ← 고도화용 새 로드맵

docs/roadmaps/ 디렉터리를 만들고 v1(MVP)·v2(고도화)로 나란히 둔다. 개발이 끝나면 완료된 로드맵 은 그대로 기록으로 남고, 다음 고도화는 새 버전 으로 시작한다. 깃의 버전 관리 처럼 — 개발의 궤적 이 그대로 쌓이는 셈이다.

로드맵 생성 — Development Planner 에이전트

새 로드맵도 맨손 으로 쓰지 않는다. MVP 때 만든 Development Planner 서브 에이전트를 ultrathink 모드로 다시 부른다.

이때 컨텍스트가 핵심 이다. 기존 prd.md 와 roadmap-v1 을 함께 참조시키면 — 에이전트가 이미 만들어진 것 위에 고도화 계획을 얹는다. 그렇게 나온 고도화 로드맵 에는 개요 → 관리자 개발 → 링크 복사 → 다크모드 까지 단계별 구현 사항 이 촘촘히 담긴다.

💡 인증은 단계적으로. 관리자 로그인은 일단 환경 변수 기반의 간단한 패스워드 인증 으로 시작한다. 회원가입까지 갖춘 제대로 된 인증은 *

추후 Supabase Authentication* 으로 — 지금은 작게, 나중에 제대로 가 원칙이다.

끝나면 — PRD도 갱신한다

고도화가 끝나면 잊지 말 것. PRD 파일도 업데이트 한다. 왜? — Claude Code가 프로젝트 맥락을 파악하는 기준 문서 가 PRD이기 때문이다.

[고도화 개발 완료]
      ↓
[roadmap-v2 완료 체크]  →  [PRD 갱신]  →  다음 버전(v3)의 든든한 컨텍스트

구현이 문서에 반영 되어야, 다음 작업에서 클로드가 현재 상태 를 정확히 안다. 코드만 바뀌고 문서는 옛날 그대로 면 — AI는 낡은 맥락 위에서 헤맨다.

정리하며

MVP 이후 로드맵 버전 관리를 정리하면 다음과 같다.

  • 마인드셋 → MVP 이후도 특별하지 않다 → 로드맵에 단계 추가 / 버전 부여
  • 고도화 항목 → 관리자 레이아웃 · 링크 복사 · 다크모드 (+자유 확장)
  • 버전 관리 → docs/roadmaps/에 v1(MVP)·v2(고도화) 나란히 → 궤적 보존
  • 로드맵 생성 → Development Planner(ultrathink) + 기존 PRD·v1 컨텍스트
  • 인증 → 지금은 환경 변수 패스워드 → 나중에 Supabase Authentication
  • 마무리 → 개발 완료 → PRD 갱신 → 다음 버전의 컨텍스트

다시 강조하지만, 이 프로젝트의 핵심은 견적서 웹의 실효성 이 아니다.

그리고 이러한 점에 초점을 맞추셔서 앞으로 수업을 들으셨으면 좋겠습니다.

Claude Code로 어떤 워크플로우를 가져갈지 — 그게 전부다. PRD → 로드맵(버전) → 전문가 위임 → 검증 → 문서 갱신 이라는 순환 만 손에 익히면, 견적서든 무엇이든 **같은 방식으로 ** 키워나갈 수 있다. 다음 챕터부터는 이 v2 로드맵을 따라 관리자 기능 을 실제로 붙여보자.

관리자 기능 개발하기

앞 챕터에서 v2 로드맵 을 짰다. 이제 계획표 를 펼쳐놓고 관리자 기능 을 실제로 붙일 차례다. 흐름은 MVP 때와 판박이 다 — 그게 바로 워크플로우가 자리 잡았다는 증거 다.

계획부터 — plan_task 는 단순 할 일 쪼개기가 아니다

언제나처럼 ROADMAP.md를 컨텍스트로 넣고 Shrimp Task Manager의 * plan_task* 로 계획을 세운다. 그런데 강사가 한 번 더 짚는 게 있다.

수강생분들이 궁금해하실 것 같아서 제가 한 번 더 설명 드리면요.

plan_task 는 그냥 일감을 나열 하는 게 아니다. Chain of Thought(사고의 연쇄)와 지능형 메모리 를 활용해 — 단순 작업 을 넘어 고도화 로드맵 전체 를 단계적으로 설계한다. 이번엔 3개의 작업 이 잡혔다.

낮은 복잡도로 — 지난 시간의 교훈

작업을 실행하기 전, 컨텍스트 용량을 확인하고 정리한다.

우선 /clear 작업으로 대화 내용을 클리어 해주시고요.

그리고 execute_task로 구현에 들어간다. 이때도 원칙은 지난 시간과 동일하다.

하지만 저는 오류를 줄이기 위해서 최대한 낮은 복잡도로 위임 을 해요. 이전 시간에 말씀드렸죠?

큰 덩어리 를 한 번에 맡기면 — 강사의 경험상 버그 확률이 확실히 올라간다. 그래서 잘게 쪼개 시키는 게 정석이다.

구현 — Next.js 전문가에게

관리자 레이아웃은 Next.js에 능숙한 서브 에이전트에게 맡긴다. 이번 챕터에서 붙이는 것들은 이렇다.

  • 관리자 레이아웃 — 대시보드 골격
  • 관리자 인증 — 로그인 페이지 (비밀번호는 .env.local 환경 변수로 관리)
  • 견적서 관리 — 목록·상세 보기

개발 중 오류가 나면? — 브라우저 테스트 로 확인하고, 서브 에이전트 안에서 곧장 고친다. 인증이 완성되면, 어드민 로그인 페이지 에서 비밀번호 한 줄 로 관리자 대시보드에 들어갈 수 있다.

오류 진단 — 로그를 그대로 클로드에게

여기서 이 강의의 진짜 핵심 인 오류 해결 루프 가 다시 등장한다. 견적서 목록 을 클릭하니 오류 가 났다. 당황할 일이 아니다.

[좌측 하단 오류 버튼] → [오류 정보 복사] → [Claude Code에 붙여넣기]
        → [Bypass permissions로 수정] → [정상 동작 확인]
  1. 화면 좌측 하단 오류 버튼 으로 에러 정보 확인
  2. 그 정보를 그대로 복사 → Claude Code에 붙여넣기
  3. Bypass permissions 모드로 수정 진행

에러 메시지 라는 가장 정확한 컨텍스트 를 그대로 넘기는 것 — 이게 컨텍스트 기반 디버깅의 핵심이다. 수정 후 목록·상세 보기 가 정상 동작하는 걸 확인한다.

마무리 — 커밋, 그리고 로드맵 갱신

기능 하나가 돌면 루틴 대로 매듭짓는다. Git 커밋 으로 지점을 박고, ROADMAP.md에 완료 체크 를 남긴다.

네, 그러면 이번 시간은 여기서 마치기로 하고요.

관리자 기능은 아직 절반 이다 — 링크 복사·다크모드는 다음 챕터의 몫이다.

정리하며

관리자 기능 개발을 정리하면 다음과 같다.

  • 계획 → plan_task = 단순 나열 X → CoT·지능형 메모리 로 로드맵 설계 (3개 작업)
  • 실행 → /clear로 컨텍스트 정리 → execute_task → ⭐ 낮은 복잡도로 위임
  • 구현 → Next.js 서브 에이전트 → 레이아웃·인증(.env.local)·견적서 목록
  • 오류 해결 → 오류 버튼 → 복사 → 클로드 → Bypass → 컨텍스트 기반 디버깅
  • 마무리 → Git 커밋 → 로드맵 갱신

새 기능을 붙여도 흐름은 똑같다 — 계획 → 잘게 위임 → 전문가 구현 → 로그로 디버깅 → 커밋. 이 리듬 이 손에 붙으면, 관리자든 무엇이든 막힘없이 얹을 수 있다. 다음 챕터에서는 남은 * 링크 복사·다크모드* 를 마저 붙여보자.

관리자 기능 개발 및 배포

앞 챕터에서 관리자 레이아웃·인증·견적서 목록 까지 붙였다. 이제 남은 절반 — 링크 복사·다크모드 를 마저 구현하고, 재배포 까지 달려 고도화 를 매듭짓는다.

남은 작업을 모두 개발 해 보도록 할게요.

남은 작업 확인 — 로드맵부터 펼친다

언제나처럼 로드맵을 펼쳐 완료 안 된 작업 을 추린다.

로드맵을 살펴볼까요? 그러면 남은 작업 을 확인해 보고요.

남은 건 링크 관리 · 다크모드 · 통합 테스트 · 최적화 다. plan_task로 계획을 잡는다.

복잡하지 않으면 — 한 번에 쭉

지난 챕터에선 낮은 복잡도로 잘게 위임했다. 그런데 이번 남은 작업들은 크게 복잡하지 않다. 그래서 Task 16~24번 을 순차적으로 한 번에 진행한다.

  • 다크모드 — 버튼 클릭 시 UI 테마 전환 (정상 동작 확인)
  • 링크 복사 — 견적서 목록에서 공유 링크 복사
  • 이메일 공유 · PDF 다운로드 — 클라이언트 전달 경로 완비

작업의 복잡도에 따라 — 잘게 쪼갤지, 묶어서 갈지 판단하는 것. 이 완급 조절 도 워크플로우의 일부다.

배포 — main에 push하면 끝

여기서 지난 MVP 배포와 결정적 차이 가 드러난다. 이번엔 Vercel 자동 배포 가 걸려 있다.

[git commit] → [main 브랜치 push] → [Vercel가 감지 → 자동 재배포]

커밋하고 main에 push 하기만 하면 — Vercel이 알아서 빌드·배포한다. 이게 CI/CD의 힘이다.

⚠️ 또 환경 변수 — 빌드 오류와 해결

그런데 빌드가 실패 했다. 로그를 까보니 익숙한 범인 이다 — 환경 변수 다.

💡 MVP 배포 때도 강조했듯, 새로 추가된 환경 변수 는 Vercel에도 따로 등록 해야 한다. 로컬 .env.local 에만 있으면, 배포본은 그 값을 모른다.

해결은 한결같다. 오류 로그를 Claude Code에 넘겨 원인을 짚고, Vercel Settings → 환경 변수 메뉴에서 누락된 값 을 추가한 뒤 재배포 한다. 빌드가 Ready 로 바뀌면 배포 완료다.

최종 검증 — 처음부터 끝까지

배포본에서 전체 흐름 을 훑는다.

  • 관리자 로그인 → 대시보드 → 견적서 조회 → 링크 복사 → 이메일 공유 → PDF 다운로드 — 전부 정상
  • 다크모드 도 잘 동작 (원하면 클라이언트 화면 에도 토글을 열어줄 수 있다)

이로써 Notion으로 관리한 견적서를 클라이언트가 확인하는 견적서 조회 시스템 이 — 고도화까지 완성됐다.

정리하며 — 그리고 워크플로우 총정리

관리자 기능 개발과 배포를 정리하면 다음과 같다.

  • 남은 작업 → 로드맵에서 추림 → 링크·다크모드·테스트·최적화
  • 완급 조절 → 복잡하면 잘게, 간단하면 묶어서(Task 16~24 순차)
  • 배포 → git push → Vercel 자동 재배포
  • 오류 → ⚠️ 환경 변수 누락 → 로그를 클로드에 → Vercel Settings 등록 → 재배포
  • 검증 → 로그인~PDF까지 전 기능 정상 → 고도화 완료
