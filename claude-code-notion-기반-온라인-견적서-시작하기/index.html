새로운 프로젝트 소개 및 오리엔테이션

지금까지 커스텀 커맨드·서브 에이전트·훅·MCP 를 하나씩 배웠다. 이제 그 도구들을 하나의 실전 프로젝트 에 엮을 차례다.

무엇을 만드나 — Notion 기반 온라인 견적서

만들 것은 간단한 업무 자동화 웹 이다. 주제는 견적서.

비즈니스를 하시는 분들은 클라이언트에게 견적서를 제공 해야 할 일이 있는데요.

그런데 워드·엑셀 로 견적서를 관리하면 골치 아프다 — 버전 관리 가 어렵고, 수정할 때마다 번거롭다. 그래서 이렇게 바꾼다.

[Notion 데이터베이스 = 견적서 관리]
        ↓ 연동
[웹 = 클라이언트가 확인]
        ↓
[PDF 다운로드]

견적서를 Notion 데이터베이스 로 관리하고, 그걸 웹과 연동 해 클라이언트가 깔끔하게 확인하고 PDF로 내려받게 하는 것이다.

사실 견적서가 핵심이 아니다 — AI 워크플로우

여기서 강사가 중요한 못 을 박는다. 이 프로젝트의 진짜 목적은 견적서 웹 도, 수익화 도 아니다.

이번 프로젝트를 통해 알아야 할 주요 내용은 바로 Claude Code를 활용한 AI 워크플로우 예요.

지금까지는 간단한 기능 위주였다면, 이번엔 복잡한 프로젝트 를 계획하고 — 효과적으로 개발 하는 실무 워크플로우 를 익히는 게 핵심이다. 그래서 강사의 당부는 이렇다.

프로젝트에 초점을 두시기보다, 어떠한 문제가 주어져도 커다란 작업을 단계별로 나누고 원활히 개발할 수 있도록 — 워크플로우 를 학습하는 데 초점을 두세요.

견적서는 연습용 소재 일 뿐이다. 한 번 익혀두면, 같은 방식으로 Notion 기반 블로그·뉴스 사이트 등 무엇이든 만들 수 있다.

이슈는 '자연스러운' 것 — 해결 과정까지 본다

한 가지 미리 알아둘 게 있다. LLM은 확률적이라, 같은 프롬프트에도 결과가 달라진다. 그래서 미리 짜둔 정답 코드 로 진행할 수가 없다.

개발을 진행하면서 이슈가 발생하는 건 매우 자연스러운 현상 이죠. 이러한 이슈를 어떻게 해결하는지 제가 모든 걸 보여드리겠습니다.

즉 이 프로젝트는 매끄러운 완성본 을 보여주는 게 아니라, 부딪히고 고치는 과정 까지 그대로 담는다. 그 바탕에는 Anthropic 공식 문서와 다양한 사례 분석 으로 다듬은 AI 개발 워크플로우 가 깔려 있다.

정리하며

이번 오리엔테이션을 정리하면 다음과 같다.

  • 만들 것 → Notion 데이터베이스 로 관리하는 온라인 견적서 웹 (+ PDF 다운로드)
  • 진짜 목적 → 견적서가 아니라 Claude Code AI 워크플로우 학습
  • 태도 → 결과물 보다 "큰 작업을 단계별로 쪼개 개발하는" 흐름에 집중
  • 마음가짐 → 확률적 특성에 따른 이슈는 자연스러운 것 → 해결 과정까지 함께 본다
  • 확장성 → 한 번 익히면 블로그·뉴스 사이트 등 무엇이든 응용 가능

이번 챕터는 본격적인 항해 에 앞선 출항 브리핑 이다. 다음 챕터부터 실제 프로젝트 에 들어가, 지금까지 배운 도구들을 하나의 워크플로우 로 엮어보자.

메타프롬프트 활용 PRD 생성 (+MVP)

오리엔테이션에서 무엇을 만들지 정했다. 이제 진짜 첫걸음 — 설계도(PRD) 를 만든다. 그런데 그냥 만들지 않는다. 메타프롬프트 라는 한 단계 위 의 기법을 쓴다.

준비 — 스타터킷 클론하고 환경 세팅

먼저 지난 시간에 만든 스타터킷을 GitHub에서 내려받는다.

git clone <스타터킷-리포지토리-URL> online-quote
cd online-quote
npm install

프로젝트 폴더명 은 원하는 대로 지정하고, npm install 로 의존성을 깔면 출발 준비 끝이다.

💡 Tip — bypass permissions. 강의 시점에 클로드 코드가 2.0 으로 메이저 업데이트됐는데, 확장 프로그램이 불안정 할 때가 있다. 그럴 땐 터미널에서 직접 실행 하거나, 모든 권한을 허용하는 bypass permissions 모드 가 편하다. 다만 회사 프로젝트 에선 주의 — 무엇이든 허용된다는 건 위험 도 함께라는 뜻이다.

PRD란 — 나와 AI가 함께 보는 설계도

PRD(Product Requirements Document) 는 프로젝트의 목적·기능·특징 을 정의한 요구사항 문서 다.

이러한 문서를 작성함으로써 우리도 프로젝트 전체 를 파악하고, Claude Code도 우리가 원하는 프로젝트의 맥락 을 파악할 수 있는 거죠.

즉 PRD는 사람과 AI가 같은 그림 을 보게 만드는 기준점 이다. 이게 흐릿하면, 클로드는 매번 다른 방향 으로 코드를 짠다.

핵심 기법 — 메타프롬프트

여기서 강의가 진짜 강조 하는 게 나온다. 메타프롬프트(Meta-prompt) 다.

메타프롬프트는 실제 프롬프트를 생성해주는 프롬프트 예요.

PRD를 바로 써달라고 하는 대신, "좋은 PRD를 뽑아내는 프롬프트"를 먼저 만들게 하는 것이다. 프롬프트를 위한 프롬프트 — 한 단계 위에서 품질을 끌어올린다. 강의의 흐름은 이렇다.

  1. CLAUDE.md를 첨부해 기술 스택 을 파악시키고
  2. docs/PRD_PROMPT.md 에 메타프롬프트(= PRD 생성용 프롬프트)를 만들게 한 뒤
  3. 그 메타프롬프트로 docs/prd.md — 실제 PRD 를 생성

추상적인 일반 PRD 가 아니라, 이 프로젝트·이 기술 스택 에 맞춘 PRD가 나온다는 게 핵심이다.

왜 MVP인가 — 작게 시작해 검증하기

PRD를 만들 때 빠지면 안 되는 개념이 MVP(Minimum Viable Product) 다. 핵심 가치를 검증 하기 위한 최소 기능 제품 을 말한다.

시장이 원하지 않는 제품 을 잔뜩 만들어 출시했다가 실패하는 경우가 많다. 그래서 MVP로 빠르게 검증 하고, 반응을 보며 점진적으로 키운다 — 이게 애자일(Agile) 의 발상이다. 견적서 웹도 마찬가지다. 온갖 부가 기능 을 처음부터 욕심내지 말고, "견적서를 보여주고 PDF로 내려받는" 핵심부터 만든다.

다 담지 마라 — 취할 건 취하고 놓을 건 놓기

생성된 PRD에는 프로젝트 개요·비즈니스/기술 목표·성공 지표 등이 빼곡히 담긴다. 그런데 전부 필요한 건 아니다.

이렇게 일반적인 문서 는 실제 개발할 때 너무 추상적 일 수 있어요. 쉽게 말해 불필요한 내용 일 수 있죠.

이유는 토큰 때문이다.

이런 것도 다 갖고 있으면 다 토큰이 소모 돼요. 모든 게 컨텍스트로 잡히기 때문이죠.

예컨대 성공 지표(KPI) 같은 건 기획 엔 쓸모 있어도, 클로드 코드 개발 엔 굳이 필요 없다. 그래서 강사의 원칙은 명확하다.

취할 건 취하고, 놓을 건 놓는 전략이 필요해요.

정리하며

메타프롬프트 PRD 생성을 정리하면 다음과 같다.

  • 준비 → 스타터킷 git clone → npm install (불안정 시 bypass permissions)
  • PRD → 목적·기능 을 정의해 사람과 AI가 같은 맥락 을 보게 하는 문서
  • 메타프롬프트 → PRD를 바로 쓰지 말고 → "PRD를 만드는 프롬프트"를 먼저 → PRD_PROMPT.md → prd.md
  • MVP → 최소 기능 으로 빠르게 검증 (애자일) — 핵심부터
  • 전략 → 모든 지표를 담지 말 것 → 토큰 절약을 위해 취사선택

PRD는 프로젝트의 첫 단추 다. 메타프롬프트로 정교하게 뽑고, MVP 관점으로 가볍게 추리는 것 — 이 두 축이 좋은 설계도의 비결이다. 다음 챕터에서는 **서브 에이전트 **를 활용해 더 실용적인 PRD를 만들어보자.

서브에이전트 활용 PRD 생성 (+프롬프트 엔지니어링)

앞 챕터에서 메타프롬프트 로 PRD를 만들어봤다. 그런데 거기엔 불필요한 내용 이 많았다. 이번엔 한 단계 더 나아가 — 서브 에이전트 로 바로 개발 가능한 PRD를 뽑는다.

복잡함을 배제하고, 바로 개발 가능한 실용적인 명세만 있는 PRD 생성 에이전트를 만들어 보았습니다.

출력 템플릿 — 결과물의 '틀'을 미리 정하기

핵심 기법은 출력 템플릿(Output Template) 이다. AI가 내놓을 결과의 형식 을 미리 정의 하는 것이다.

저는 AI를 활용해서 강의 커리큘럼도 부분적으로 제작하는데요.

이때도 "이런 틀로 뽑아줘" 라고 형식을 못 박으면, 매번 같은 품질 의 결과가 나온다. PRD도 마찬가지다 — "무엇을 생성하지 말 것" 까지 명시해, 추상적인 군더더기 를 처음부터 잘라낸다. 이게 프롬프트 엔지니어링의 핵심이다.

prd-generator — 요구사항을 품은 전문가

메타프롬프트와 서브 에이전트의 차이가 여기서 갈린다.

방식 특징
메타프롬프트 매번 PRD 생성용 프롬프트를 새로 만들어야 한다
서브 에이전트 요구사항이 에이전트 안에 이미 박혀 있어 곧장 호출 하면 끝

prd-generator 서브 에이전트는 "어떤 PRD를 어떤 틀로 뽑을지" 를 이미 품고 있다. 그래서 메타프롬프트를 따로 만들 필요 없이, 부르기만 하면 된다.

시간은 조금 더 걸리지만, 그만큼 더 정교하고 똑똑하게 진행합니다.

서브 에이전트는 자기 컨텍스트에서 일하므로, 메인보다 느려도 더 깊은 분석 을 내놓는다. 특정 도메인 분석 이나 복잡한 로직 일수록 빛난다 — 예컨대 야구 게임 PRD 라면 "야구 룰 분석 전문가" 에이전트를 붙이는 식이다.

도메인 전문가 — Notion API 전문가 에이전트

이 프로젝트엔 Notion API 연동 이라는 도메인 지식 이 필요하다. 그래서 전용 에이전트를 하나 만든다.

  • 역할 — 웹에서 Notion API·데이터베이스를 잘 다루는 전문가
  • 모델 — Opus (깊은 분석이 필요하니 강한 모델)
  • 모드 — 플랜 모드 (코드는 안 건드리고 분석·요약만)

이 에이전트가 Notion API 사용법 을 분석·요약해두면, PRD가 현실적인 명세 위에 올라선다.

만들어보기 — 그리고 훅 알림

이제 prd-generator 를 bypass permissions 모드로 호출해 PRD를 생성한다. 서브 에이전트는 시간이 걸리니, 이럴 때 **훅 알림 **이 빛난다 — 기다리는 동안 다른 일을 하다, 끝나면 슬랙으로 받는 것이다.

완성된 PRD는 확실히 다르다.

  • 핵심 요약 — 군더더기 없이 두 줄 로
  • 사용자 여정 — 견적서 입력 → 클라이언트에게 URL 전달 흐름을 상세히
  • 핵심 명세 — 표 로 정리, 누락 방지 를 위해 ID 값 까지 표기

⭐ 한 번 더 — PRD Validator 로 교차 검증

여기서 강의가 진짜 중요한 습관 을 보여준다. PRD 검증 이다.

경험이 있으신 분들은, 실제 Meta가 지원하는 Instagram API 를 들어가 본 후에 — 이게 구현 가능한 건지 검토를 해봤을 거예요.

AI는 "인스타 릴스 업로드 API" 처럼 실제로는 안 되는 기능을 되는 것처럼 PRD에 적어버릴 수 있다. 이를 막는 게 PRD Validator 서브 에이전트다.

  • 역할 — PRD 기술 검증 전문가 → 실제 구현 가능한 레벨 인지 검사
  • 원칙 — 환각 방지·사실 검증 → API 기능을 추측하지 않는다 고 명시적으로 선언

즉 생성 에이전트 가 PRD를 쓰면, 검증 에이전트 가 크로스 체크 한다. 만드는 AI ≠ 검증하는 AI — 이 분리가 환각으로 쌓은 명세 를 걸러낸다.

우리가 원하는 프로덕트를 만들 때는 첫 단추를 잘 끼우는 것 이 정말 중요하거든요.

정리하며

서브 에이전트 PRD 생성을 정리하면 다음과 같다.

  • 출력 템플릿 → 결과의 틀 을 미리 정의 → 군더더기 없는 실용 명세
  • prd-generator → 요구사항을 품은 에이전트 → 메타프롬프트 불필요, 곧장 호출
  • 도메인 전문가 → Notion API 전문가(Opus·플랜 모드)로 현실적 명세 확보
  • 생성 팁 → 시간이 걸리니 훅 알림으로 완료 통지
  • ⭐ 검증 → PRD Validator 로 구현 가능성 교차 체크 → 환각 방지

좋은 PRD는 잘 쓰는 것 만큼 잘 검증하는 것 이다. 생성 → 검증 두 에이전트로 첫 단추 를 단단히 끼웠으니 — 다음 챕터에서는 이 PRD를 바탕으로 실제 개발 에 들어가보자.
